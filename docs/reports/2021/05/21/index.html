Creator:
James R. Griffin III
Created:
2021-05-21 14:07:07
Status:

Comments:
James R. Griffin III at 2021-05-21 14:12:42 :

In order to advance this, a number of steps were required:

  • Ensuring that the bindings for port 8091 were enabled for the TLS on the Microsoft Server environment from within the IIS configuration
  • Ensuring that the user under which the application is running has the proper access control list privileges to bind to this port
  • Ensuring that Elements provides access (at the application level) to the API endpoint for clients authenticated over the VPN

The second step for this is achieved with the following Powershell command (invoked with administrative privileges):

PS C:\windows\system32> netsh http add urlacl url=https://+:8091/elements-api/v4.9/ user=%USERDOMAIN%\%USERNAME%
James R. Griffin III at 2021-05-21 14:16:59 :

HTTP Basic authentication is then offered in the response from the server for the API endpoint:

curl -v https://oaworkflow.princeton.edu:8091/elements-api/v4.9/
[...]
>
< HTTP/1.1 401 Unauthorized
< Content-Length: 0
< Server: Microsoft-HTTPAPI/2.0
< WWW-Authenticate: Basic realm="Symplectic Elements at Organisation (API)"
< Date: Fri, 21 May 2021 14:14:52 GMT
<
* Connection #0 to host oaworkflow.princeton.edu left intact
* Closing connection 0

Please be aware that this does not integrate with Single Sign-On for the application.

#4529 Workers stop picking up jobs

Creator:
hackartisan
Created:
2021-05-21 17:44:29
Status:
Open
Comments:
Trey Pendragon at 2021-05-21 21:29:21 :

TL;DR: We need to wait for one of these to break to get a thread backtrace to figure out if we have a thread leak.

https://app.honeybadger.io/projects/53391/faults/35108866/01F65P9QDDF1EEW4EEEB18WG9B is the error that's killing these. As soon as this happens that worker's dead and can no longer pick up jobs.

According to https://github.com/mperham/sidekiq/issues/2862 this is likely because there's a thread leak in our code somewhere - something's spawning a thread and not getting rid of itself, we'll have to make sure that thread-spawner's not run in the Sidekiq process. Maybe Bunny? That'd be my guess.

We can find out by waiting until a worker dies, leaving it, and doing the following:

In one pane:

sudo tail -f /var/log/syslog | grep sidekiq

In another:

ps aux | grep sidekiq (get sidekiq Process ID) Check the thread count: sudo ps HuH p sidekiq_pid | wc -l <-- This should be pretty high if it's our problem. sudo kill -TTIN sidekiq_pid

Then copy the backtrace of all the threads (dumped into syslog) into this issue so we can diagnose them. The TTIN signal will dump the backtrace, as per https://github.com/mperham/sidekiq/wiki/Signals#ttin

The OTHER possibility is https://github.com/rails/rails/issues/33600.
