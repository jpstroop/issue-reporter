Background: The Derrida Seminars collection includes materials that were printed on reused paper, with unrelated correspondence on the backs of the documents. This creates a very disjointed reading experience.

#4529 Workers stop picking up jobs

Creator:
hackartisan
Created:
2021-05-21 17:44:29
Status:
Open
Comments:
Trey Pendragon at 2021-06-10 21:23:37 :

Today we found out:

The worker systems are running out of available file handles. The reason this is happening is because doing Valkyrie::StorageAdapter.find_by(id: "") initializes a file handle to the given file on disk. There's no way to close that handle, and Sidekiq is threaded so the process never goes away. So every file that's ever opened stays opened. Garbage collection seems to never clear up that reference, probably because of something in Valkyrie.

f = File.open("/opt/repository/files/c8/dc/7c/c8dc7cba826940c1af8807052fb12da6/4f16c4243.tif")
# Open in lsof = 3
f = nil
GC.start
# Open in lsof = 0

f = Valkyrie::StorageAdapter.find_by(id: "disk:///opt/repository/files/c8/dc/7c/c8dc7cba826940c1af8807052fb12da6/4f16c4243.tif")
# Open in lsof = 3
f = nil
GC.start
# Open in lsof = 3

This seems to be because the IO is part of an instance variable, so GC doesn't grab it?

class NewFile
   attr_reader :io
   def initialize(io)
     @io = io
   end
end

f = NewFile.new(File.open("/opt/repository/files/c8/dc/7c/c8dc7cba826940c1af8807052fb12da6/4f16c4243.tif"))
# lsof count = 3
f = nil
GC.start
# lsof count = 3
Trey Pendragon at 2021-06-10 22:09:45 :

Apparently Valkyrie can register a finalizer to close the IO stream if it's garbage collected.

Edit: I can't get this to reliably work in a testable way in Ruby 2.x

Trey Pendragon at 2021-06-10 22:30:23 :

So I think Valkyrie needs to:

  • Provide access to #close
  • Not open a file descriptor until you're trying to read the content of a file.
