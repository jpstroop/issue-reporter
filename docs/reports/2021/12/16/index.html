#553 Update the oaworkflow-dev and oaworkflow X.509 certificates

Creator:
James R. Griffin III
Created:
2021-11-23 18:13:29
Status:
Closed
Comments:
James R. Griffin III at 2021-12-16 21:52:46 :

This was not necessary to address within the dev server environment, however, several steps were needed in order to resolve this issue:

  • LibreSSL must be built from source (I have used LibreSSL 2.8.3 within a VM environment for Ubuntu 14.04 LTS)
  • The private key and X.509 certificate (including the intermediary and root CAs) must be bundled in the PKCS#12 format, and exported with a password (set when openssl is invoked in the shell)
  • The PKCS#12 must be imported into the Internet Information Services entry for the windows server. If the wrong version of LibreSSL or an incompatible version of OpenSSL is used, then this will raise an error.
  • The X.509 must be specified as a binding for TCP/UDP traffic over the port 443. There may also be errors here, should the key not be extracted in the Windows Server environment (this was the case for the production environment, but not the dev environment)
  • With admin. privileges, one must then invoke > certutil -repairstore -user My $CertSerialNumber. $CertSerialNumber is found within the Serial Number field of the response for certutil -store -user My
  • Then, one should be able to successfully bind this fixed X.509 certificate for traffic over port 443.
