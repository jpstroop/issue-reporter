#1616 Slow loading of reserve locations

Creator:
Christina Chortaria
Created:
2021-08-18 18:02:41
Status:
Open
Comments:
Hector Correa at 2021-08-18 19:31:50 :

Some background information on how the Availability calls to bib data and Alma work in this scenario.

The current Orangelight code makes one call to bibdata to get information about availability (https://bibdata.princeton.edu/bibliographic/availability.json?deep=true&bib_ids=9960099173506421) and the response looks as follows:

{
    "9960099173506421": {
        "fake_id_1": {
            "on_reserve": "Y",
            "location": "firestone$res3hr",
            "label": "Firestone Library - Circulation Desk (3 Hour Reserve)",
            "status_label": "Unavailable",
            "copy_number": null,
            "cdl": false,
            "temp_location": true,
            "id": "fake_id_1"
        }
    }
}

Notice that because the item is in a temporary location Alma returns no ID for the holding (hence the "fakeid1" in the response).

Since we don't have a valid holding id we make a second call to bibdata (https://bibdata.princeton.edu/bibliographic/9960099173506421/availability.json?deep=true) that forces bibdata to call the Alma items API to get the information. The response for this second call looks like this:

{
    "9960099173506421": {
        "22695150500006421": {
            "on_reserve": "Y",
            "location": "firestone$res3hr",
            "label": "Firestone Library - Circulation Desk (3 Hour Reserve)",
            "status_label": "Not Available",
            "copy_number": "1",
            "cdl": false,
            "temp_location": true,
            "id": "22695150500006421"
        }
    }
}

Notice that this response has the holding id (22695150500006421) rather than the fake placeholder. However, this second call is slow since the call to the Alma items API takes 2-3 seconds (on top of the 1second that the first call took!)

Some takeaways from this:

  • Our use of the deep=true is a bit confusing. Notice that both calls to bibdata pass deep=true but only the second one is slow because calling /bibliographic/{id}/availability.json?deep=true follows a different path than /bibliographic/availability.json?deep=true&bib_ids={ids}. My bad, mostly.

  • Indexing the actual (temporary) location into Solr might be a good idea since we will be able to display to the user that information as soon as the page render. But, we might still need to make a second (slow) API call to Alma to get the actual holding_id. This might be transparent to the user, though.
