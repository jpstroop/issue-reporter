#284 Create list of empty bitstream filenames

Creator:
Kevin McElwee
Created:
2021-03-09 19:16:01
Status:
Closed
Comments:
Kevin McElwee at 2021-03-10 21:50:11 :

This was quite the investigation!

The query select * from metadatavalue where resource_type_id = 0 and text_value IS NULL; returned no results for both the staging and dev server.

The query select * from metadatavalue where resource_type_id = 0 and text_value = ''; showed 1 result on the staging server, and 0 results on the dev server. Lynn corrected this entry. (Staging, Dev)

Working backwards from the example provided revealed that the problem wasn’t filenames that were NULL, but bitstreams that didn’t have a dc.title field, which is resource_id 64. On the prod server there are 619 items that fall in this category. They've been placed into this google sheet: https://docs.google.com/spreadsheets/d/1LhDt9ppgX_Q3uCqKhr3HZieLq12-Up1DlmQW5JrjxF8/edit?usp=sharing

Here is the query: sql SELECT handle,bitstream_id FROM handle INNER JOIN item ON (item.item_id = handle.resource_id) INNER JOIN item2bundle ON (item.item_id = item2bundle.item_id) INNER JOIN bundle2bitstream ON (item2bundle.bundle_id = bundle2bitstream.bundle_id) WHERE bitstream_id IN ( SELECT resource_id FROM metadatavalue WHERE resource_type_id = 0 GROUP BY resource_id HAVING NOT (64 = ANY(ARRAY_AGG(metadata_field_id))) );

I'm moving away from this for now, but I would imagine we might want to figure out what our next steps are? I can include collection / community information so that we know what's exclusively Lynn's concern? Some of the metadata includes import information, from which we can grab filenames, so fixing this may be automatable.

Side note: There were 635 on dev, 20 on staging, and (as mentioned above) 619 on prod. How exactly are dataspace databases being synced?
