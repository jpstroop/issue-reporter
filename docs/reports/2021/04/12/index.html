#887 Eliminate case sensitivity in hyper-linked search functionality (Keywords, for example)

Creator:
Kim Leaman
Created:
2021-03-31 16:29:30
Status:
Open
Comments:
hackartisan at 2021-04-12 13:52:28 :

@tpendragon observed that keywords weren't included in the default search

hackartisan at 2021-04-12 14:52:40 :

The functionality here is provided by blacklight's link_to_search. I really think this is the intended behavior. It's actually being renamed link_to_facet. Facet values don't provide the kind of string processing that other types of fields do because it's expected that they be very precise. Generally a data cleanup would be used to resolve this type of problem. If keyword was selected as a facet value for this collection the faceted search results will also suffer from this same issue until the data is cleaned. @tpendragon do you have a different perspective?

Trey Pendragon at 2021-04-12 20:04:06 :

It seems like the standard way to fix this is to present facets from a field without the lowercase, but facet via a field with lowercase. I'm not sure if Blacklight can display differently, and I don't think we're indexing into a field which lowercases but doesn't tokenize.

Options to fix:

  1. Clean up the non-lowercased keys.
  2. Do 1, but also make it so keyword lowercases on save.
  3. Add a special field type in solr which lowercases but doesn't tokenize, and index keywords into there.

2 is probably easiest and most complete?

hackartisan at 2021-04-12 20:13:32 :

And then also capitalize them (sentence case) on presentation, I assume. Since most of them are capitalized.

This seems like a reasonable solution.

#787 UV issues: not displaying MVWs, information/contents bars properly for some PCs

Creator:
Kim Leaman
Created:
2021-03-31 15:49:47
Status:
Open
Comments:
Trey Pendragon at 2021-04-12 21:53:59 :

I fiddled a little for a quick solution and there isn't really one.

  1. The viewer shrinks to mobile mode if the width of the container is < 805 px or so. This happens in our layout at about 1350px, which is higher than the XL bootstrap breakpoint even.

  2. Full screen doesn't even load the thumbnails, unless the page had thumbnails on first initialization. I assume that pane just doesn't render if the size is too small. (If you resize the window and full screen you get thumbnails, but if you resize the window and refresh, no thumbnails)

To fix this I think we either have to:

  1. For XL screens the table of contents collapses to the left with a little three line pop-out icon
  2. Find some other layout
  3. Add a thumbnail view to the mobile version of UV.
  4. Switch to Mirador for everything, and just accept that MVWs are going to look bad.
