#203 Troubleshoot and resolve Solr reindexing issues for the DataSpace staging environment

Creator:
James R. Griffin III
Created:
2020-10-23 16:08:50
Status:
Open
Comments:
V. Alex Brennen at 2021-04-19 13:22:03 :

This is typically a file permissions problem (specifically a lock file) problem with your solr search index directory.

If you're running a solr instance for your DSpace install on a separate server, you should verify that that system is running correctly and that the solr instance can read and write to the directory where the search collection for your dspace system is located.

If you're running a solr instance for your DSpace install locally with in the same Tomcat (or Jetty) instance (like most people), you need to do the same thing. In this case, your solr search index (and other indexes) would be located in ${dspace.dir}/solr. For search specifically, ${dspace.dir}/solr/search and for the actual index files ${dspace.dir}/solr/search/data/index . Make sure that Tomcat can write to these directories and create new files. Typically, file ownership permissions, where the Tomcat/DSpace user cannot write to the ${dspace.dir}/solr directory and its subdirectories is the problem.

If file permissions are not the problem, you may have stale lock files. Solr, in the default DSpace installs, defaults to "native" locking. However, it includes options in its configuration to use "simple" (lock file existence) locks and even to clear all locks on start-up. It also includes "single" locking that should not be used with DSpace. Simple locking is fine for DSpace in my experience. However, using the "clear all locks" option as part of your regular configuration can be dangerous.

If you suspect a stale lockfile problem, shutdown Tomcat and remove all files named "write.lock" in all subdirectories of ${dspace.dir}/solr . Then, restart Tomcat, and attempt to update your search indexes (bash$ dspace index-discovery). Note, if you did not ever successfully create your search indexes in the first place, you should probably run index-discovery with a '-b' flag which will "(re)build index, wiping out current one if it exists".

If you still see file locking issues and are sure your file permissions/ownership are correct, try switching your solr collections to "simple" locking. You can find this setting in ${dspace.dir}/solr/(collection)/conf/solrconfig.xml. You can also find the "clear all locks" setting in this file. To apply it, shutdown Tomcat, remove all write.lock files, and restart Tomcat. Then attempt to update your search indexes.

If this still doesn't work, try setting "clear all locks" for all your solr collections, shutdown tomcat, remove any write.lock files, and restart tomcat. Then, try running index-discovery again. Note: If this does work, you'll want to shutdown tomcat afterward and disable the "clear all locks" setting.

Finally, if none of this works, it's possible that your DSpace home directory isn't being set properly and that the solr webapp for DSpace is therefor not configured with proper directory location for the solr directory. If that is the case, it wouldn't be able to find the indexes in order to open or update them. This is configured in your local.cfg, your dspace.cfg, and in your solr webapp directory at ${tomcat.home}/webapps/solr/WEB-IMF/web.xml file. Setting in your local.cfg should be sufficient. The value stored in web.xml is taken from the dspace.cfg file in src/dspace/config/dspace.cfg during the build process. You can directly set your solr home in that web.xml file in the object. However, if the web.xml file is incorrect, the web.xml files would be incorrect for all the DSpace webapps.
