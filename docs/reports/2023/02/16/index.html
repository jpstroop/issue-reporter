#45 Middleware/API architecture proposal

Creator:
Chuck McCallum
Created:
2023-02-16 21:59:26
Status:
Open (pull request)
Comments:
Esmé Cowles at 2023-02-16 22:31:09 :

I thought I'd share a couple of thoughts and questions about how this would work:

  1. To what extent would Rails controllers fulfill the role of the API (i.e., the views would constitute the UI)? Would scrupulously creating controller endpoints for any data that controllers or views/helpers fill the gaps?
  2. I've seen token-based auth folded into controllers so browsers and automated clients use the same endpoints, but I think it's more common to have different endpoints.
  3. One of the ramifications of having a significant external command-line/API customer base that Matt, Kate, and I discussed recently is that any backwards-incompatible changes would potentially break their integrations. Maybe this argues against considering Rails controllers the API?
  4. What are the best practices for having a separate, versioned API?
  5. I'm not sure that having a "Middle Layer" between the UI/API and the backend is a requirement, or if it's one possible approach to making sure that users don't have unmediated access to MediaFlux just because they are using a command-line tool or API integration. Basically, maybe this app is the "Middle Layer"?
