#5958 Wireframe user optimized Virtual Reading Room path options

Creator:
Trey Pendragon
Created:
2023-08-16 16:10:23
Status:
Open
Comments:
Shaun Ellis at 2023-10-17 16:31:11 :

Some more thoughts on the Aeon approach:

If a user enters their email address, we can check Aeon via the API to verify that they're registered, create a "Request" for them (that goes into a custom VRR Queue), and provide instant access. We can also query all VRR requests by a single user, and we can also query all the VRR requests in the Custom Queue, which can be filtered by date. This is all through the Aeon API, so we can create a separate interface for querying VRR requests, presumably to get statistics/analytics (i.e., what resources are popular) or to investigate abuse/use that violates our policies.

All this is to say that if we go with one of the other workflows, we will need to build a backend that supports a queryable record of user accounts, the user's registration date, date of acceptance of terms, and a list of VRR content ids that they requested. Since our terms may change, we will need to be able to re-ask the user to accept the terms if they have not done so since the change.

One final note is that implementing the feature that checks if a user is already registered via Aeon would greatly improve the Request Workflow of the current Finding Aids site as well because it would not require the user to login and leave the content page before they can submit a request.

The one issue I see with the Aeon approach is that there is no way to register a user without sending them to the Aeon site to create their account. If there was a way to redirect them back to the referring page, this option would be both easiest to implement and a good user experience, in my opinion.
