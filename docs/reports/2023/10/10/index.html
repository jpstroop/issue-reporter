#5978 Repeatably Generate a Report on the Cloud Preservation Status of All Resources

Creator:
Trey Pendragon
Created:
2023-09-19 20:27:42
Status:
Open
Comments:
Trey Pendragon at 2023-10-10 15:00:16 :

We ran this on production and found most resources failed because metadata checksums didn't match.

This is a flaw in checking metadata checksums as an idea. We add fields to FileSet over time, so the serialization changes. For instance, we've added service_targets. So if a FileSet is ingested on October 1, it says the following on both the cloud service and locally:

{
"title" => ["Test"]
}

but if we then add service_targets on October 3, locally it now says:

{
"title" => ["Test"],
"service_targets" => []
}

. The file's still preserved - if we restore the old file into the new it works, we're backwards compatible. However there's no way to translate the checksum for the old file serialization to the new.

We might want to change preservation metadata validation by storing the lock token on the object's metadata in GCS and checking that. We'd have to change our preservation function too.

hackartisan at 2023-10-10 20:14:38 :

Another idea is to not include properties that don't have values in the metadata json. This seems a bit simpler.

Either option will require re-preserving all the metadata targets.

Note: This process is not fully automated. There may be a delay before we respond to you with credentials or a demo request after you complete this form.
