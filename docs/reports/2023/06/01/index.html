#5845 Investigate Adding InCommon as an Authorization Provider

Creator:
Trey Pendragon
Created:
2023-05-24 16:12:42
Status:
Open
Comments:
Trey Pendragon at 2023-06-01 16:47:28 :

I met with Keith Martin today from IAM. Summarizing some information here:

  • Keith recommends we start by replacing our OAuth via CAS with SAML2 using Princeton's identity provider. He says @kevinreiss would be a good partner to walk us through how that works, and I have a memory of @carolyncole writing some docs about how to do that.
  • We'd have to register Figgy as a "service provider" with InCommon. We'd be the first published service provider for Princeton. This is pretty simple and fast - there are some encryption requirements for our certificate, and we have to be using SAML2 for auth, but Keith can get us the details. We'd also have to provide some contact information and stuff.
    • This costs some money. Keith will figure out how much, and who would pay the bill.
    • We haven't done this before, but Keith's very excited about finding opportunities to not use guest accounts, from a security perspective.
  • We'd have to implement an "identity provider picker" that lets folks choose from a list of institutions to log in from. Then the remote identity provider would have to accept our service provider - some accept all registered InCommon service providers, some don't. Keith says Simon Lee has done some proof of concept here that we can look at.
    • Keith says he doesn't expect this will be a lot of work to keep up. If folks try to log in and it fails they'll contact their local IT, their local IT will see it's an InCommon service, and they'll easily be able to allow it (or contact us or Keith)
