Jane Sandberg at 2023-03-03 23:12:35 :

We aren't always guaranteed to be able to find the more general language name (the "macrolanguage" using ISO 639-3 terminology) given a certain ISO 639-3 code.

The provided example, wuu, is included in the ISO 639-3 macrolanguage table with a macrolanguage of zho (Chinese). In the example MARC record, the 008 contains the ISO 639-2B code chi (Chinese), so we could also generate the requested facet "Chinese" from that.

However, another example: nuz (Tlamacazapa Nahuatl) isn't mapped to a macrolanguage of Nahuatl, since no such macrolanguage exists in ISO 639-3 (it does in ISO-639-2 and ISO-639-1). It seems that we would want both "Tlamacazapa Nahuatl" and "Nahuatl" in the facets -- will CaMS catalogers always put in both the ISO 639-3 and 639-2B code in these records?

Jane Sandberg at 2023-03-03 23:00:58 :

Some possible gems:

  • https://github.com/bbenno/languages -- it does not currently include macrolanguages and has not yet had a version 1.0.0 release, but does have CI and a nice script for updating the code data.
  • https://github.com/mlj/iso-codes -- on version 1.2.0, tests but no CI, does include macrolanguages

We could also forego using an additional gem, and use the debian iso-codes package or the tables directly from SIL.

Jane Sandberg at 2023-03-03 23:14:16 :

Also noting that, while we currently use the language_facet field for both the facet and the show page, we'll have to use two separate fields (and do the appropriate orangelight work) in order to meet these requirements.

#3441 Can't download MARC file for record

Creator:
rladdusaw
Created:
2023-03-03 19:06:52
Status:
Open
Comments:
mzelesky at 2023-03-03 19:37:11 :

Extremely long contents fields cause many other problems as well. This doesn't impact the ticket, but I am planning on sweeping through all records periodically to locate fields that are longer than 9,999 bytes.

The byte limit for an individual field in the MARC standard is 9,999 bytes. The byte limit for a record is 99,999 bytes.

There is an option in the ruby-marc gem to allow oversized records, but I don't know if that would really solve this (since it would result in an invalid MARC file).

Perhaps you could check the record before serving it up to see if the record is too large to export in binary format?

This is code I use to find problems like this:

def record_size?(record)
    total_size = 0
    total_size += record.leader.bytesize if record.leader
    record.fields.each do |field|
      total_size += 1 # field separator character
      total_size += field.tag.bytesize
      if field.class == MARC::ControlField
        total_size += field.value.bytesize
      else
        field.subfields.each do |subfield|
          total_size += subfield.code.bytesize
          total_size += subfield.value.bytesize
          total_size += 1 # subfield separator character
        end
      end
    end
    total_size > 99999
  end

  def field_size?(record)
    return true if record.leader.bytesize != 24

    record.fields.each do |field|
      if field.class == MARC::ControlField
        return true if field.value.bytesize > 9999

      else
        total_size = 0
        field.subfields.each do |subfield|
          total_size += subfield.code.bytesize
          total_size += 1 # subfield separator character
          total_size += subfield.value.bytesize
        end
        return true if total_size > 9999

      end
    end
    false
  end
