#5669 Local fixity only runs on primary file

Creator:
hackartisan
Created:
2023-02-09 14:09:07
Status:
Open
Comments:
Trey Pendragon at 2023-03-28 17:39:48 :

@eliotjordan A file can have a PreservationFile and an IntermediateFile, and both of those are hand-crafted files. If one of them disappears we can't get them back. PreservationFile is -more- important, but still.

For instance, hand-placed watermarks is one use case. We'd have to try to re-hand-place watermarks if we lost that intermediate file. Another is hand-crafted audio derivatives.

Edit: Preservation/Intermediate is documented here: https://github.com/pulibrary/figgy/blob/main/docs/ux/use_cases.md#preservationfile--intermediatefile

Trey Pendragon at 2023-03-28 19:14:51 :

Eliot and I looked at this together, and found the following things:

  • We don't calculate local checksums for IntermediateFiles. Characterization only runs on primary_file, and then the derivative services look at intermediate after that.
  • Cloud preservation re-generates the checksum for every binary node it preserves the first time, which includes intermediate nodes. This means the cloud has a checksum value for preserved intermediate files, and is able to check that.

To close this ticket we'd have to do the following:

  • Calculate checksums for intermediate files during characterization (maybe re-queue characterization when appending an intermediate file?)
  • We'd have to run characterization on every FileSet that has an intermediate file.
  • We'd have to store the child_id for local fixity events as the FileMetadata ID it ran against (or change the meaning of a failed local fixity check to be "one of the binary files failed figure it out")
  • If we do the above, we'd have to re-run local fixity on everything again. We'd also have to delete all the old local fixity events.
