#5391 Arabic section titles are not appearing as Right To Left in Universal Viewer

Creator:
Shaun Ellis
Created:
2022-08-24 19:29:38
Status:
Open
Comments:
hackartisan at 2023-05-04 13:23:52 :

https://github.com/pulibrary/figgy/pull/5750 updates manifests as per the implementation proposal in the previous comment. However, UV only implemented the rtl direction feature for the metadata info pane, not for the leftPanel "Contents" iiif-tree-component.

Code references:

  • the TreeView in the leftpanel module uses TreeNode for the elements of the table of contents https://github.com/UniversalViewer/universalviewer/blob/86f2cd47598ed2b2cef6b30fd3dc706b0c8b9807/src/content-handlers/iiif/modules/uv-contentleftpanel-module/TreeView.ts
  • The TreeNode itself is defined in manifesto: https://github.com/IIIF-Commons/manifesto/blob/6e355e34c38b64b9cf9c366d6ff60006db341968/src/TreeNode.ts
    • the label is set in the constructor, making it much more difficult to override the behavior here, to the extent that it's either not actually possible or really not a good idea.
  • Here's some code showing how it works in the metadata info pane: https://github.com/UniversalViewer/universalviewer.github.io/blob/85dac1a4274f53659ba136c2bcd7ce1ab8c9d908/vendor/uv/lib/iiif-metadata-component.js#L375-L382
  • language direction configuration referenced in the UV ticket: https://github.com/UniversalViewer/universalviewer/blob/67ada49f6d866a51ab173d7166f6ddf04fffb471/src/content-handlers/iiif/extensions/uv-mediaelement-extension/options/en-GB.json#L50

Note that @tpendragon also tried just putting html as the manifest value, but that resulted in an error.

The proposed path forward is (not necessarily in this order): * [ ] Work on UV upstream to implement the required feature, possibly in collaboration with someone in the UV community. * [x] Migrate from webpacker to vite (https://github.com/pulibrary/figgy/issues/5734) * [ ] Update figgy to use the latest UV release, ensuring we carry over local customizations (see #5778)

However, this path forward is a bigger project than we had hoped and the work will need to undergo prioritization again.

#5528 Redesign the preservation status indicators on the resource show page

Creator:
Trey Pendragon
Created:
2022-11-17 20:29:21
Status:
Open
Comments:
Shaun Ellis at 2023-05-04 20:56:47 :

Given that there may be several types of errors on a given fileset, that preserved filesets should display "preserved", and cards should remain the same size, here is a revision. The errors and warnings stack on top of each other below the thumbnail. They could potentially obscure the thumbnail if there are enough. I removed the color of the card outline that corresponds to the error or warning since they can be mixed.

One thing to note is that I'm assuming DLS is not notified of a fixity failure and that the user must contact DLS to resolve it. I think the easiest thing to do here is to allow the user to report it, sending a message with a unique identifier to a DLS queue so we can try to resolve the issue manually. Once it's reported, it displays "Reported" and is no longer actionable.

We will need to display the following error messages:

  • Repairing Derivatives
  • Repairing Local Fixity
  • Repairing Cloud Fixity
  • Derivatives Failed | Report
  • Local Fixity Failed | Report
  • Cloud Fixity Failed | Report
