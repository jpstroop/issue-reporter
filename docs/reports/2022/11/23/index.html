More analysis: EphemeraFolder state readability is as permissive as possible: If it's containing box has a readable state, it will treated as having readable state no matter what its own state is, and if it's containing box doesn't have a readable state, its own state will be used to determine.

https://github.com/pulibrary/figgy/blob/150e9def951fd0b1ea8f948069f5a0225fff4f4f/app/decorators/ephemera_folder_decorator.rb#L212-L221

However, the mechanism that powers this relies on a hack that codes "needs_qa" as a readable state. Then lots of other code relies on that readable state array.

readable state definition: https://github.com/pulibrary/figgy/blob/150e9def951fd0b1ea8f948069f5a0225fff4f4f/app/models/folder_workflow.rb#L30

Code relying on that array: * https://github.com/pulibrary/figgy/blob/28ce5082801c4e62a6cc1d2992e3aa8372f73194/app/decorators/valkyrie/resource_decorator.rb#L183-L188 * https://github.com/pulibrary/figgy/blob/6b66dc52aa4187a2a2d0d2e2fb087cefd35f3e1e/app/presenters/permission_badge.rb#L37-L40 * https://github.com/pulibrary/figgy/blob/c5f851aea6e7c65306e17fb40ec9cf74c40a0a7f/app/change_set_persisters/change_set_persister/mint_identifier.rb#L30 * https://github.com/pulibrary/figgy/blob/9ac0402b4255b356f83360c5453a6b49d0ce1602/app/models/ability.rb#L203-L218 * * https://github.com/pulibrary/figgy/blob/9ac0402b4255b356f83360c5453a6b49d0ce1602/app/models/ability.rb#L120-L123

All the ephemera folders currently say they will sync to discovery sites, regardless of the state of their containing box. Here's one example with a "shipped" box: * folder: https://figgy.princeton.edu/catalog/b43b32bd-874b-415a-8350-75e553342d42 * box: https://figgy.princeton.edu/catalog/626af16f-152f-4985-a106-70afd3d80320

I'm not sure if they actually are syncing to discovery sites. More investigation needed on that point.

It seems like we might want to change this so that the public read states are accurate, and update child state at save time the way we do with other propagations. This might necessitate a migration.
