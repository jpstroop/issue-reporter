On Mediaflux Developer Guide p. 8:

DIME Encoding • NOTE: Support for DIME has been depreciated in Mediaflux. This documentation exists purely for historical reasons Mediaflux uses Direct Internet Message Encapsulation (DIME) encoding for all messages. Knowing about DIME is only relevant if any of the following applies: • You are using web-services, in which case DIME encoding must be specified when setting up the web-service client application. • You need to write a native client in a language other than Java (and cannot use web-services).

Further down:

The native format is used for TCP/IP transport. Native Mediaflux messages are DIME encoded. The first record in the DIME message contains a Mediaflux service request or response, described below. The subsequent DIME records contain attachments, if any.

Reading https://en.wikipedia.org/wiki/Direct_Internet_Message_Encapsulation

… the limit being the size information for each chunk, which was 32 bits so 1 gigabit …

(ie, the zero bytes are there because the file length was less than 2^16… if it were longer, there would be fewer zeros.)

So there seem to be mixed messages here: It’s deprecated… but it’s used for encoding all messages?

… and there is a Ruby library for it: https://formats.kaitai.io/dime_message/ruby.html

(But maybe we shouldn’t go too deep here? My sense is that we’re probably not going to be supporting upload through the app … just moving around files that are already on disk? But it will still probably come in handy… :shrug: )
