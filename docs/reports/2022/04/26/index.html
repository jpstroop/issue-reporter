#4774 Resources updated in Pulfalight should update their metadata in Figgy automatically.

Creator:
Trey Pendragon
Created:
2021-11-05 19:52:46
Status:
Open
Comments:
Trey Pendragon at 2022-04-26 00:06:12 :

Historically we've used a pull model for this - now all of our sources are us and under our control in some way. I'd like to use a push model so that remote services tell Figgy "hey this changed". That way if Figgy's down we can leave it to the other applications to retry.

Options I see:

  1. /identifiers/:identifier/update - Notify Figgy an identifier is updated.
  2. GraphQL mutation
  3. GraphQL pub/sub
  4. WebSub (https://www.w3.org/TR/websub/#h-note-0)

Something else...?

Whatever we do, we should probably allow the API to accept multiple identifiers being updated. Maybe even wildcards? So when a collection call number is updated it gets all of the components.

Eliot Jordan at 2022-04-26 13:58:59 :

I vote for options 1 and 4. GraphQL doesn't seem appropriate for this use case. WebSub is interesting.

Shaun Ellis at 2022-04-26 14:47:01 :

My colleague at the Princeton Development Office used WebSub in 2008 (then called "PubSubHubbub") to sync a number of our applications. I definitely think the Pub/Sub model is the way to go for this type of use case, so I'd say #4 seems most appropriate, and allows us to easily sync future systems with the same infrastructure.

Not sure what is involved in #1. Is the idea that it's just an API endpoint that waits for some system to send it an ID to then update?

I'm also a bit wary of GraphQL pubsub as I feel like we may have instances where we will want to sync systems in which GraphQL is not involved. It also seems a bit underdeveloped based on the docs.

I assume each of these solutions will also require some plugin/code to be written for ASpace that "publishes" data updates (or in the case of #1, makes an API request), correct?

Trey Pendragon at 2022-04-26 15:12:19 :

I assume each of these solutions will also require some plugin/code to be written for ASpace that "publishes" data updates (or in the case of #1, makes an API request), correct?

We update from Pulfalight - my intention was to just have Pulfalight ping Figgy after an index job.

Trey Pendragon at 2022-04-26 15:13:49 :

For 4 I'd have to look at the spec more than the cursory glance I gave it.

How would we handle Figgy having to be both the "Hub" and "Consumer" if we went that route?

hackartisan at 2022-04-26 17:08:06 :

Option 1 seems quickest to implement. Do the benefits of pubsub outweigh that consideration? Assuming pulfalight runs a background job, and the figgy endpoint also kicks off a job, I think we have a very low potential to lose these events.

Would we ever have other subscribers?

Trey Pendragon at 2022-04-26 20:50:08 :

We grew consensus around #1.

Open questions:

  • [ ] An entire finding aid updates at once, which includes potentially thousands of Component IDs. Do we send every one as its own request? Do we allow wildcards? Do we have a different endpoint for "everything which is part of this archival collection"?
  • [ ] What's the API route?
hackartisan at 2022-04-26 21:18:18 :

I like the idea of a wildcard, also fine with a separate endpoint. this seems like a case where on the figgy side it might be worth looking into using async for the network IO, so I don't think we should send each as its own request.
