#559 Allow retrieving suppressed bib records

Creator:
Esmé Cowles
Created:
2019-09-06 16:09:37
Status:
Open
Comments:
Christina Chortaria at 2022-09-01 22:09:20 :

@kevinreiss this is a very old issue when we were still using voyager_helpers with voyager. As @mzelesky posted we allow retrieval of suppressed bibs through the barcode endpoint only. I think we can close this issue there has been some discussion in the past see https://github.com/pulibrary/bibdata/issues/710#issuecomment-670607954 and https://pulibrary.slack.com/archives/CA891BNR0/p1600875040001900.

Esmé Cowles at 2022-09-01 22:51:19 :

The main use case that motivated this ticket is Figgy's remote metadata feature. We have had cases where content was digitized but could not be linked to the Bib records because they were suppressed, which made the workflow very convoluted and involved having to put the Bib ID into the title field and then move it to the source metadata identifier field later once the records were unsuppressed.

I think we plan to make Figgy use Orangelight instead of Bibdata to retrieve metadata at some point (@tpendragon is this right?). But I'd like to keep this ticket until we've done that or found another solution.

Trey Pendragon at 2022-09-01 23:22:34 :

That's right and it's mostly there. Presumably getting suppressed records from the orangelight endpoint would be much harder.

Kevin Reiss at 2022-09-02 16:07:43 :

Yes, suppressed records would not be present in Orangelight at all. You'd need to go directly bibdata to get the metadata. We'd need to make an endpoint that would respond with suppressed records by MMS ID lookup. How critical is this feature? Could figgy default to trying OL and if the MMS ID lookup failed go to another endpoint that is able to provide suppressed record data?

#368 Required field validation in Works is broken

Creator:
Hector Correa
Created:
2022-09-01 18:28:40
Status:
Open
Comments:
carolyncole at 2022-09-02 16:26:06 :

I would work on this but I can not. Can someone be more specific about this? If we want to user to be able to save before they add the description in the first screen of the wizard, then we can not just make the description a required field at all times for a work to be saved. Please clarify the requirements.

Hector Correa at 2022-09-02 19:17:05 :

I will be out next week on PO training so here are my thoughts on this:

In general I think we should let Work records be saved without the required fields while the Work is on the draft state. We will enforce required-fields validation as soon as the user completes the Wizard (i.e. we don't let the work move from draft to waiting for approval until all required fields have been entered). From that point forward we always enforce the required field validation.

This will allow a user to come and go from the Wizard at their leisure (e.g. they might not know what kind of license they want to use, but they will still be able to upload files). They might leave the Wizard with some incomplete fields and come back later once they know what license or description they want to use.

Currently we do not allow the user to move from the very first step of the Wizard until they have provided a title and a creator. But the user is free to move through the rest of the Wizard even if the other fields have not been provided. I think this is OK.

One thing that we should change is that we could send the user to the Wizard anytime they edit a work that is still on draft state. Right now, we only send them to the Wizard the first time, but never again. I think it will be more intuitive to the user to always see the same interface (the Wizard in step 1, not step 0) while they are in draft. Once the Work is in any another state (i.e. waiting for approval) then we could use the normal edit form since I don't think curators probably won't care about or need the Wizard.
