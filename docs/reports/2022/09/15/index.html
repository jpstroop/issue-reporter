Creator:

Status:
Open
Comments:
Bess Sadler at 2022-09-15 12:29:04 :

@carolyncole Great question. I have gone back and forth on this. When I wrote that guide to syncing S3 buckets I was picturing that we would make a new Globus service, spin up an empty S3 bucket for it, and then synch the data. However, the more I understand about our projects and the state of the legacy data the more I lean toward migrating the files one by one. The legacy Globus S3 bucket contains lots of empty folders named with DOIs. That's going to make it very hard to tell when we've correctly migrated any given work. I now believe that a better workflow would be, for each object:

1) Describe the record in pdcdescribe 2) Submit the data payload via pdcdescribe a) via the form for objects < 100Mb b) upload directly via Globus or S3 for big data 3) A curator checks the DataCite record that we produce and refines as necessary 4) Upon approval, the data payload is moved to the new S3 bucket and deleted from the legacy location a) This will also require indexing into PDC Discovery and updating the download link there

QUESTION: Should there also be a step where we manually delete the migrated object from DataSpace? I'd like to, if possible, because it will make it more clear that PDC Describe is the canonical copy, and it would prevent potential snarls where we've already migrated a record but then someone goes in and makes changes to the DataSpace record and those changes aren't persisted to PDC Describe. However, it's hard to advocate for deleting a record from DataSpace before we have very good assurance that PDC Describe is a fully operational production system with reliable backup and restore. So maybe that's a thing to prioritize?

Alicia Cozine at 2022-09-15 21:52:37 :

I did a little digging on this one because I didn't understand it. Noting what I learned here: You can set these values in Postgresql. You can also set these values in the operating system, in /etc/sysctl.conf. The two sets of settings have slightly different names. - Time of inactivity before first keepalive request is sent, in seconds: - Postgresql: tcpkeepalivesidle - Sysctl.conf: tcpkeepalivetime - Time between one keepalive request and the next, in seconds: - Postgresql: tcpkeepalivesinterval - Sysctl.conf: tcpkeepaliveintvl - Max number of keepalive requests sent before connection is considered broken: - Postgresql: tcpkeepalivescount - Sysctl.conf: tcpkeepaliveprobes

Default value for all settings in Postgresql is 0 which picks up the OS setting. Consensus seems to be that it's best to set these in sysctl.conf. Is this something we want to set only on psql boxes, or across all infrastructure? @hackartisan's suggested settings would keep each connection alive for 5 and a half minutes (30 seconds before keepalives begin, then 10x30 seconds of keepalive requests) - does that seem like the right overall time?
