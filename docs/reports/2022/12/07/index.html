#3592 Ensure only 1 PreservationObject is made per Resource.

Creator:
Trey Pendragon
Created:
2020-01-16 20:53:36
Status:
Open
Comments:
Eliot Jordan at 2022-12-06 22:08:28 :

The most recent PreservationObjects created for the same Resource resource are linked to this Coin: https://figgy.princeton.edu/catalog/6c6994a7-4abd-4b6b-8488-38b128fb70c2

The PreserveResourceJob was queued 5 times in a 4 minute span.

Theory A Coin is created and marked complete. Changes are made and saved multiple times before any other additional PreserveResourceJobs have completed. A new preservation object is created here on each job invocation: https://github.com/pulibrary/figgy/blob/1c3e9ae220157e4294b7fde634e0dfec8d27f713/app/services/preserver.rb#L78

This would presume that the Coin has no member FileSets because the PreserveResourceJob is run synchronously when a Resource has members.

https://github.com/pulibrary/figgy/blob/d7d7e9bc945a4c96d483d7e6d1cc173f9fcd6c5a/app/change_set_persisters/change_set_persister/preserve_resource.rb#L20-L21

hackartisan at 2022-12-07 15:11:48 :

Sounds like a convincing theory!

Trey Pendragon at 2022-12-07 19:24:30 :

Maybe when we queue up a PreserveResourceJob we should pass its optimistic lock token and compare before we do it?

Eliot Jordan at 2022-12-07 19:40:01 :

@tpendragon Am I understanding your suggestion?

  1. Pass the lock token value of the resource to be preserved as a param of in the PreserveResourceJob
  2. When the job is run, compare that param value with the current value of the resource.
  3. Stop the job if they don't match

Seem like a good idea, although optimistic locking is only enabled for FileSets and ResourceChargeLists. Will prevent most of the multiple PreservationObject instances, but not all.

Trey Pendragon at 2022-12-07 22:19:40 :

I think they're populated on the other resources, just not verified. Worth checking? It was an implementation detail
