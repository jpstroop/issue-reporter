#229 Describe and migrate this record: Sowing the Seeds for More Usable Web Archives

Creator:
Bess Sadler
Created:
2022-07-23 15:47:45
Status:
Open
Comments:
Hannah H at 2022-08-08 17:45:45 :

Note

Please reach out to me before implementing a change in metadata for the legacy records. In addition to differences between the schema, some liberties were taken with applying Dublin Core and there are some changes in how we want to optimally describe assets moving forward.

Description

The following table shows Dublin Core from the record in this ticket and the complementary Datacite property. The Datacite property will link to the part of the schema 4.4 help document that discusses the specific property.

Some Dublin Core will not map neatly to Datacite. I have noted properties that I'm seeking further input on. I will update the table and notify RDSS when I have gathered more complete information.

Dublin Core Datacite Notes
Title Title
Contributors Creator In the context of this example, the "contributors" are "creators" of the work. Note that an additional field for "contributor" in Datacite regards non-authoring contributors of the deposit.
Issue Date PublicationYear This is the year the object was published to the repository. Hector has made this automatic for the Describe Form.
Related Publication placeholder HH looking into nuance of this.
Abstract Description, with sub-property descriptionType; suggest "abstract" from controlled list for sub-property This will be something to discuss further, because "abstract" is too suggestive of the associated paper's abstract for a depositor to always understand what we are asking for, which is a unique description of the deposited data, code or other object. For the new form and metadata, we want to encourage "Description" to capture this information, but may want to optionally allow a choice of description types, if someone wants to get more specific. Please reach out if you want to talk about this.
Description Description, with sub-property descriptionType; suggest "other" from controlled list for sub-property The blurb entered to this field is fine for a supplementary description, though it wouldn't stand as a good description on it's own. If this record didn't also contain the "abstract" then a summary description should be taken from the README (if the submitter is unavailable), because we need a good description that will include searchable terms to be indexed by search engines. Additionally, an end-user shouldn't have to access the README to understand if this deposit is what they are looking for.
URI
  • For a DOI minted with Datacite: Identifier
  • For an ARK: AlternateIdentifier with mandatory sub-property
 Some legacy records may only have an ARK, but new records (and some legacy) will have the DOI minted via Datacite that should be used with Identifier.
Appears in Collections placeholder To be discussed further with Wind who has concerns about continuing to organize our records in this manner. It may still be useful categorization, but we might not make this as prominent as before.
