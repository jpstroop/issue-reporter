The element with div id="view" is always at the bottom of the page. The viewer-container is the id of the iframe viewer container. We are building this element with id='viewer-container' in javascript. When there is a thumbnail with icon content indicating there is a viewer at the bottom of the page this thumbnail is part of an element with href 'viewer-container'. When the user clicks on the thumbnail or content-icon it will navigate them to the viewer-container at the bottom of the page. example of an element is: <a href="#viewer-container"><div class="document-thumbnail has-viewer-link" data-oclc="[&quot;785740708&quot;]" data-bib-id="9970446223506421"><span class="sr-only">Go to viewer</span><img alt="" src="https://iiif-cloud.princeton.edu/iiif/2/c8%2Ff6%2F71%2Fc8f67121600746e6aa6ddfd1c3fe1d20%2Fintermediate_file/square/225,/0/default.jpg"></div></a> When there is a text-anchor 'Digital content' in the record page with a link having the fragment https://catalog.princeton.edu/catalog/9946018083506421#view this will still navigate the user to the bottom of the page , because the div element with id=view is always at the bottom of the page and exists as a wrapper of the 'viewer-containers'. Unfortunately this cannot happen right now because the catalog domains that are being created through the indexing process are pulling ids from the figgyarkcache which is very old and containes voyager ids. This is why currently in Digital Content : the url would be 'https://catalog.princeton.edu/catalog/4601808#view'. When the user clicks on the url it will open a new page because it resolves to the alma id. The figgyarkcache cannot be updated because the specific rake task is broken. This is a different ticket and another discussion but all are connected https://github.com/pulibrary/orangelight/issues/3104. @maxkadel I'm looking into this ticket as well as the rest that are connected in #3104

#42 Enhance Japanese translation of OJS

Creator:
Kate Lynch
Created:
2022-05-26 19:10:43
Status:
Open
Comments:
Kate Lynch at 2022-08-04 19:55:35 :

Upon extensive examination, I've found the following:

  • The missing translations for this issue are already in the Japanese language translation maintained by Weblate, but they are not in the code.
  • Translations are pulled from files in both the locales directory at the root of the application folder, and in the lib/pkp/locales directory.
  • The "Custom Locale Editor" plugin allows any existing translation to be edited by the user, in the admin interface. However, the files must already be present on the server, and the translation key must also already be present in one of those files in order for the user to edit it.
  • Custom locale information added in the admin interface is stored in the database; there is a "restore defaults" option for locales buried in the website settings under "Hosted Journals/Settings Wizard/Journal Settings/Languages" where you expand the language in question and click "reload defaults." See attached screenshot (because this was pretty hard to find)
  • Updated locale translations (pulling from the latest Weblate-translated versions by the community) are tracked by PKP in version control in a separate repository from OJS, https://github.com/pkp/pkp-lib. These contain substantially more locale files than what comes with the OJS tarball typically used for installation. The pkp/pkp-lib repository is tracked as a git submodule within the greater OJS project (https://github.com/pkp/ojs/blob/main/.gitmodules#L2).
  • In this case, there were substantially more Japanese language translation files at https://github.com/pkp/pkp-lib/tree/main/locale/ja_JP than in the base installation. Adding the manager.po file explicitly under lib/pkp/locales/ja_JP in the staging installation did in fact appear to address the missing translation. This is obviously not sustainable, however it should be noted that finding a sustainable way to update the locales from pkp-lib is likely a reasonable approach to address this.
  • For local development, it is recommended that developers clone the OJS repository and initialized submodules to get all of the latest code. It is unclear to me what the recommended approach is for non-custom installations.
