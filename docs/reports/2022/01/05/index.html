#439 Sorting by date is not working

Creator:
Esmé Cowles
Created:
2020-01-29 13:15:51
Status:
Open
Comments:
Hector Correa at 2022-01-05 21:12:13 :

When sorting by date (e.g. "sort by year, newest first") the system is sorting based on the pub_date_sort field in Solr. The exact sort clause is pubdatesort desc, title_sort asc.

The results displayed to the user come in the correct order based on the pub_date_sort value stored in Solr, however, what we display to the user sometimes shows either no date or a different date than the one stored in Solr.

For example, item 2591 has a pub_date_sort in Solr of 1821 but the TEI file for this item has a date of 1819 and we display the 1819 date to the user. You can view these values here: https://cicognara.org/catalog/2591.json, look for fields tei_date_display and pub_date_sort.

To a user this record should be displayed along other items from 1819 but instead if shows mixed with records from 1821 because the pub_date_sort has the 1821 value. See screenshot below.

Notice that when we display a record page (or a search result item) we are NOT using the data in Solr, instead we are using the pre-generated partials views under app/views/pages/catalogo. For example, for record 2591 we display the partial https://github.com/pulibrary/cicognara-rails/blob/main/app/views/pages/catalogo/_item_2591.html. Notice that this partial has the 1819 date.

We could probably re-generate the partials to include the pub_date_sort rather than the tei_date_display (or in addition to it?) but I am not sure of the ramifications of this.

Esmé Cowles at 2022-01-05 21:22:08 :

Basically, it looks like the problem is that the display date and the data from the MARC records don't match. Maybe we should sort on tei_date_display, since that's pulled from the TEI like the displayed date?

Hector Correa at 2022-01-05 21:41:39 :

The data in `teidatedisplay' is a mess, sometimes it has values like "A. 1541," or "1813 al 1818," or multiple values.

Plus it is stored but not indexed field.

Esmé Cowles at 2022-01-05 21:55:36 :

@hectorcorrea OK, so it won't be as easy as that. But taking a look at the TEI, it does look like the overwhelming number of items have a four-digit year (or a range like 1813 al 1818). Could we update the indexer to parse the first 4-digit number into a new indexed field for sorting?
