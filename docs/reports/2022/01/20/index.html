Creator:

Comments:
Hector Correa at 2022-01-20 20:26:57 :

@escowles @kelynch I will be interested in your thoughts on the solution proposed on this PR. The only issue that I believe could cause confusion to users is that sometimes we render the "related names" as links and sometimes as text (i.e. not links.)

As indicated in the PR we render them as text (i.e. not links) when the values are not indexed in Solr, but this might be confusing to users. Yet, I could not come up with a better approach.

We could always render them as text (never as links) if we wanted to prevent confusion but then we lose some functionality since sometimes the data in the Book is just as good as the data in the Version record.

Another approach would be to index the data in the Version records but that would make this much more complicate problem.

Esm√© Cowles at 2022-01-20 20:32:02 :

@hectorcorrea It makes sense there isn't always overlap, because of the different data sources. I do think it would be somewhat strange to have items sometimes be links and sometimes not.

I like the idea of indexing the values from the Version records, since that would let us make them always be links, and would probably improve searching functionality in general. But I'm not sure what the complexity (and performance) impact of that would be. Is the indexer already loading those records, or would indexing these fields require a lot more data loading/parsing/etc.?

Hector Correa at 2022-01-20 20:40:57 :

@escowles The Version information is stored in the Version table but not indexed.

We could try to update in place the Solr documents (i.e. only the related_names_display field) for the Books documents with the information on the Version records as they are processed. I don't think the performance will be an issue, I am mostly concerned on how long it will take me to figure it out since I am not very familiar with the code base. I'll take a stab at it and report back tomorrow.

The Characterization job was running on

fileset: https://figgy.princeton.edu/catalog/0c9a91d4-9549-422f-8193-83cff9d4f691, title: 9919354213506421-commentary.pdf parent: https://figgy.princeton.edu/catalog/be16e338-1b8a-43f9-a4d9-83ed17032157 parent's parent (MVW): https://figgy.princeton.edu/catalog/19c83d65-e3fc-4219-8883-f14f3ef4e811

The file is the preservation file pdf. All page images appear to have generated as desired and the viewer looks good. The parent items both say 54/55 files processed.

With these pdfs, they are initially ingested as original files, and they are changed to preservation files during derivative generation https://github.com/pulibrary/figgy/blob/b19cd2d19a69147783bb67c5a5282083eecfc007/app/derivative_services/pdf_derivative_service.rb#L132

This implies that the error here occurred due to someone attempting to regenerate derivatives for this pdf. This is something we want to allow, and the proposed fix of using primary_file in the NullCharacterizationService should fix it. Once that's in place we should be able to regenerate derivatives on this item to clear the error that displays on the parent resource pages.

#954 dash vs dot in older figgy records not showing online content

Creator:
Christa Cleeton
Created:
2021-12-21 14:34:07
Status:
Open
Comments:
Trey Pendragon at 2022-01-20 22:48:57 :

So the ID in Pulfalight was changed in the URL, but the actual call numbers still have periods in them.

For example: https://aspace.princeton.edu/staff/resources/2161#tree::archival_object_733628 The call number for this is AC107.07_c897. If you put that in Figgy it works fine.

In fact, if you use the "-" version (which is only in the pulfalight URL) in Figgy then I think DAO updating wouldn't work, since it'd try to find the archival object and not find it (there's no "AC107-07_c897").

So I think there's a few things to do here:

  • [x] Update all instances of "-" in Figgy with "." (this should only be new records I think?)
  • [x] Update the Pulfalight javascript that looks up the viewer to use periods instead of dashes.
  • [ ] If a user enters a "-" and it finds a record whose actual call number is "." it should use that as the source metadata identifier.
