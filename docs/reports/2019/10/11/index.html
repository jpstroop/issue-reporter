Eliot Jordan at 2019-10-11T17:07:39 :

This ticket proposes to preserve "everything", but that is a little misleading. A few thoughts...

  • We should not preserve resource types that are only nested within other resources. Correct?
  • The existing preservation logic only preserves resources that are "complete"., however there are a number of resource types without a state property. We will need to change the logic for these. I assume it's preserve on save when there is no state.
  • There are handful of resources that I don't know if we should preserve or not (see below).

Esmé Cowles at 2019-10-11T17:22:32 :

@eliotjordan I think it might make sense to break down by a couple of different categories of resources: 1. "Works" and collections, e.g., ArchivalMediaCollection, Collection, EphemeraBox, EphemeraFolder, MediaResource, NumismaticCoin, NumismaticIssue, Recording, ScannedMap, ScannedResource, and VectorResource. The most convenient way to identify these is the resources with a state property, where we can preserve them when they are complete. 2. Nested resources — I assume these are never preserved because they are preserved in their parent resources. 3. Helper resources, e.g., most of the numismatic, ephemera, and controlled vocab fields, which should be preserved on every save. 4. The preservation machinery itself, like Event, PreservationObject, Tombstone. I don't know if these need to be preserved (maybe they need to be preserved so we can access historical versions if we do a complete restore?) — but there may be complications in preserving them and preventing recursive preservation processes. 5. FileSet is probably a special case — needs to be preserved, but only as part of a parent "work".

Trey Pendragon at 2019-10-11T17:28:16 :

Regarding 2., nested resources are preserved individually - it's just that their test for "should I be preserved" is "does my parent say it's preserved"

James R. Griffin III at 2019-10-11T20:28:43 :

The Ansible documentation on variables can be found here: https://docs.ansible.com/ansible/latest/user_guide/playbooks_variables.html

Specifically, the variable precedence follows this scheme:

  • command line values (eg “-u user”)
  • role defaults
  • inventory file or script group vars
  • inventory group_vars/all
  • playbook group_vars/all
  • inventory group_vars/*
  • playbook group_vars/*
  • inventory file or script host vars
  • inventory host_vars/*
  • playbook host_vars/*
  • host facts / cached set_facts
  • play vars
  • play vars_prompt
  • play vars_files
  • role vars (defined in role/vars/main.yml)
  • block vars (only for tasks in block)
  • task vars (only for the task)
  • include_vars
  • set_facts / registered vars
  • role (and include_role) params
  • include params
  • extra vars (always win precedence)

(please see the section https://docs.ansible.com/ansible/latest/user_guide/playbooks_variables.html#variable-precedence-where-should-i-put-a-variable)
