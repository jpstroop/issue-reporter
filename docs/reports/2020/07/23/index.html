#324 Define a process for managing releases

Creator:
James R. Griffin III
Created:
2020-07-23 20:17:52
Status:
Open
Comments:
James R. Griffin III at 2020-07-23 20:28:45 :

Prior to this discussion, releases were based upon specific branches addressing individual GitHub issues. Branch-based deployment was undertaken into the staging environment where the proposed changes were tested and reviewed, followed by the opening of a pull request. When a pull request was merged into the main branch, the main branch would then be deployed to the staging environment.

James R. Griffin III at 2020-07-23 20:32:28 :

One proposed alternative to this approach would be the usage of tags in order to track a potential release. A predetermined set of issues would be resolved by pull requests, merging commits into main, after which a tag would then be created. This tag would then be deployed into the staging environment, and manually tested by following a defined quality assurance process.

Should this release candidate be pass the acceptance criteria defined for this QA process, then the tag would be deployed into the production environment.

James R. Griffin III at 2020-07-23 20:33:47 :

Thus far, I have only been using GitHub Milestones in order to classify Issues which are scoped for the Minimum Viable Product (MVP) or 1.0 releases. Perhaps this would be an appropriate time to attempt to utilize GitHub releases instead of Milestones?
