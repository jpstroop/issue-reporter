#305 ASpace does not import/export our ark url's pointing from agents to records

Creator:
regineheberlein
Created:
2020-07-15 15:44:51
Status:
Open
Comments:
regineheberlein at 2020-07-15 15:54:44 :

Also need to investigate the best way of importing/handing to Lyrasis our current CPF files. ASpace creates agent records based on names found in the EAD. There is a lot of variety in the string values of the names as represented in the EADs that have already been merged into single identities in the CPF records. Not all of them have viaf or loc id's, so I expect ASpace to create too many duplicate identities. Will test.

A workaround may involve putting back a consolidated name string into the EAD based on matching cid (from resourceRelation in our CPF records) combined with the name string.

See also https://github.com/pulibrary/archivesspace-development/issues/29

regineheberlein at 2020-07-15 16:11:07 :

Current urls are based on the EAD ark and the cid. The EAD ark exports from ASpace, but cid's are overwritten by aspace id's.

regineheberlein at 2020-07-15 17:20:27 :

Links created internally by ASpace do export, e.g. 

 <relations>
      <resourceRelation resourceRelationType="creatorOf" xlink:href="https://aspace-dev.princeton.edu//repositories/5/resources/1464" xlink:type="simple">
        <relationEntry>Blair and Lee Family Papers</relationEntry>
      </resourceRelation>
    </relations>
regineheberlein at 2020-07-15 17:25:11 :

This suggests a workaround where we put our CPF entities back into the EADs and import them as part of the EADs, which will trigger the internal ASpace link creation. Testing suggests that those links will then export as part of the CPF record.

For that to happen, we need to normalize the string values of the names across the EADs to prevent duplicate entities from being created.
