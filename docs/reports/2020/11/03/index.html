Shaun Ellis at 2020-11-03 21:26:20 :

@tpendragon I think I can describe it, and these rules go all the way up to the collection level:

  • When a parent component's children are all restricted (aka, <accessrestrict type="closed">), the component should get a "Restricted Content" indicator.
  • When a parent component's children have at least one restricted child, the component should get a "Some Restricted Content" indicator.
  • When a child component has restricted content, it should get a "Restricted Content" indicator.

I recently heard about <accessrestrict type="review"> in which I believe we haven't yet reviewed the content for access restrictions, but will upon request. Perhaps @apferrar , @faithc, or @regineheberlein can weigh in on what should be shown here... "Possible Restricted Content"?

Trey Pendragon at 2020-11-03 21:29:36 :

@sdellis Thanks! I've copied your comment to the root issue and moved it back over from blocked.

regineheberlein at 2020-11-04 00:19:59 :

@sdellis @tpendragon The only values currently in our EADs are "open", "closed", and "review". This was changed by ADAPT fairly recently.

In ASpace, both "review" and "closed" will be set to "restrictions=>true" in ASpace ("open" will correspondingly be "restrictions=>false"), with the text of the restriction stored in "notes". On the topcontainer record, the default value for the type of restriction will most likely be a generic "Restricted"--we're still in the process of working that out with Lyrasis--and the text of the note I believe will be in "activerestrictions".
