#1263 Provision solr 8 boxes for use by catalog-staging

Creator:
hackmastera
Created:
2020-01-31 15:46:08
Status:
Closed
Comments:
hackmastera at 2020-02-26 21:15:21 :

Migrating solr data to a new znode.

  1. On each solr box, create a copy of /etc/defaults/solr.in.sh to /etc/defaults/newsolr.in.sh. Update the ZK_HOST param to append the new znode. When it's time to edit this config file, you can just swap in the copy.
  2. Stop solr on all boxes.
  3. On one box only:
    1. Start a tmux session.
    2. cd /opt/solr
    3. bin/solr zk mkroot /solr7 -z lib-zk-staging1:2181,lib-zk-staging2:2181,lib-zk-staging3:2181
    4. bin/solr zk cp -r zk:/clusterstate.json zk:/solr7/clusterstate.json -z lib-zk-staging1:2181,lib-zk-staging2:2181,lib-zk-staging3:2181

For dsc[2] containers, this should be fairly straightforward. All retrieval containers are described in the dsc[2], e.g. MC016 Box 26: <c level="otherlevel" otherlevel="physicalitem" id="MC016_i2"> <did> <container type="box">26</container> <unitid type="barcode">32101106254731</unitid> <physloc type="code">rcpph</physloc> </did> </c> Items or subcontainers contained in the retrieval container are referenced in the dsc[1] through the container/@parent, e.g. MC016 Box 26 Reel 6: <container type="reel" parent="MC016_i2">6</container>

The situation is a little murkier for "classic" containers because it is not always evident whether a container is the retrieval container or a subcontainer.
