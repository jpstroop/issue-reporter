#822 Update catalog_date field in traject for Alma records

Creator:
Christina Chortaria
Created:
2020-09-29 13:58:36
Status:
Open
Comments:
Christina Chortaria at 2020-12-07 15:14:44 :

  • [ ] Single Records https://github.com/pulibrary/marc_liberation/issues/925 refs https://github.com/pulibrary/marc_liberation/issues/748

  1. Get a single alma record with p_avail and e_avail parameters: https://developers.exlibrisgroup.com/console/?url=/wp-content/uploads/alma/openapi/bibs.json#/Catalog/get%2Falmaws%2Fv1%2Fbibs%2F%7Bmms_id%7D

  2. If it has any AVA fields that correspond to p_avail (physical availability), get the list of items to get the <creation_date> . Get the earliest creation_date from the list.

  3. If there are no AVA fields that correspond to p_avail, look for AVE fields that correspond to e_avail. Each AVE field has a portfolio id included. For each AVE field in the API response, you must retrieve the portfolio information. Use the oldest <portfolio><activation_date> from the attached portfolios. See https://developers.exlibrisgroup.com/alma/apis/docs/xsd/rest_portfolio.xsd/

  4. If there is no p_avail or e_avail, use <bib><created_date>.

Christina Chortaria at 2020-12-07 15:22:04 :
  • [ ] Publishing profiles https://github.com/pulibrary/marc_liberation/issues/926

Bibliographic Enrichment -> Create date subfield 950b Physical Items Enrichment -> Create date subfield 876d Electronic Inventory Enrichment -> Activation date subfield 951w

  1. Similar to the Single Record if there is a 876 field don't use the 951 fields.
  2. If there are no 876 or 951 then use the bib create date: Bibliographic Enrichment 950b.

PROBLEM: Don Thornbury just alerted me to the fact that the AbID db is also used by catalogers / non-archivists, for data that does not live in ASpace. (They use MacroExpress to add it to Voyager). That means the current implementation plan doesn't work, since it relies on ASpace to know what the next available number in the sequence is.

Essentially, to maintain that functionality, we need a freestanding db that both generates and stores the AbIDs (or the last available for each profile, anyway), and then EITHER overwrites the ASpace record OR outputs the AbID's as XML (or sends them to Alma, later).

(In light of this, I wonder whether it's easier to just keep the current Access db, and just add functionality to import its XML output into ASpace.)
