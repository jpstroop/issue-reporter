#213 Allow requesting of components with level="otherlevel" otherlevel="text"

Creator:
faithc
Created:
2020-03-31 20:40:12
Status:
Open
Comments:
regineheberlein at 2020-03-31 20:53:36 :

Best practice is to put container information on the terminal/leaf-level component. This best practice is based on the assumption that the terminal component represents a retrievable physical item. However, we have exceptions where we have encoded components at the "sub-item" level (e.g. prayers in a prayer book, pages of a volume, photographs pasted into an album). These are identified with c level='otherlevel' otherlevel='text'.

I see two issues being brought up here: 1. Requesting currently looks for container information only on the terminal level. That means that requesting for these components is not enabled. (The logic being that you can't request a piece of an integral item, nor can you request something non-physical like text.) To address this, a condition is needed that, when an "otherlevel" component is encountered, looks up to the first ancestor that does have container information. (We may need to exclude dao-only (i.e. digital) components in this.) 1. In addition @sdellis is addressing the question of metadata in figgy, which pulls from the EAD component, i.e. does not show container information for "otherlevel" components. A similar condition may be needed here.

Trey Pendragon at 2020-03-31 20:59:56 :

Can I propose we rename this ticket (in order to be actionable) to Allow requesting of components with level="otherlevel" otherlevel="text"?

Edit: Also, can we not ship without this? Has the original site ever been able to do this? I'm not saying it's a ton of work, I really don't know, it's just a smaller footprint of required features makes for a quicker release date, and a quicker release date usually results in easier prioritization and recognition of issues.

faithc at 2020-03-31 21:31:44 :

I think renaming the ticket per your suggestion makes sense.

The fact that these items aren't requestable on the current site was deemed to be a bug, so it would be great if we could address the requesting functionality at least as part of the mvp. As of now, the majority of items in the collection given as an example, C0744.04, are not requestable. This is why public services staff brought it to our attention.

Trey Pendragon at 2020-03-31 21:34:23 :

Cool, thanks! 😄
