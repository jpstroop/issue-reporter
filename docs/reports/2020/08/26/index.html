#4221 why aren't the individual items for grounds and buildings showing up in figgy?

Creator:
Kim Leaman
Created:
2020-08-25 20:31:51
Status:
Open
Comments:
Trey Pendragon at 2020-08-25 22:34:55 :

The volumes are added to the collection, but the MVWs aren't. Search results hides volumes (because you'd get a bunch of "Vol. 1", etc.) Usually the MVW is in the collection, but not the volume.

I should have mentioned - want me to just take all those items out of a collection and put their MVW in one?

Esmé Cowles at 2020-08-26 15:22:43 :

This is really a continuation of https://github.com/pulibrary/figgy/issues/3815 — and I think we have a couple of options for how to move forward here, in approximate order of how difficult I expect they would be to do:

  1. Sync the MVWs over to DPUL and deal with them at that level (and use ordering and structure for description within the MVWs)
  2. Add some logic to figure out that we want to sync the members of these MVWs (not sure if there's a way to identify those now, or if we'd add a new property, or something else)
  3. Revisit the modeling and consider making these collections instead of MVWs (which would solve the DPUL problem, but bring back the problem from #3815) — we have several cases of photo albums and photograph collections with similar issues, so being able to do this would help more than just this collection
  4. Revisit the decision to avoid item-level description of these items in the finding aid
Kim Leaman at 2020-08-26 16:13:30 :

Thanks, @tpendragon and @escowles. Ok, so: 1. we could, but is order/structure really appropriate for yet more description if we are using it already as a discovery tool for each item in the MVW? I dont see it as ideal or working with every use case. I may be wrong. 2. Can we talk a bit more about this possibility? 3. If it helps: as a work around, we were able to get around the photograph collections in Western Americana by treating the photo album itself as an individual work with a MARC record. all the individual photographs are still FA items. 4. This seems the more reasonable way forward, since this is a big reason why we are revisiting problems with ingest and metadata for more than one collection at this point.

Trey Pendragon at 2020-08-26 16:25:18 :

Oh I see. You want the items individually discoverable in DPUL. Yeah, number 4 please. Number 2 we'd have to add a way to say "yeah these are members but make them discoverable and make them sync"

Kim Leaman at 2020-08-26 16:28:52 :

Good to know and thank you! I'll try my best, @tpendragon. @escowles and @cwulfman let me do a test to see how adding the extra stuff to order/structure may look and we can go from there for this particular project, just to get the migration done for it and forthcoming postcards. standby
